# FINAL_IMPLEMENTATION.md
## HumanMaximizer AI Lead Generation & Research Assistant

---

## Executive Summary

An AI-powered B2B lead generation pipeline built for the HumanMaximizer HRMS sales team. The system accepts a business keyword, runs a three-agent LangGraph pipeline (Research → Qualification → Sales), retrieves product knowledge from ChromaDB indexed against humanmaximizer.com, generates RAG-grounded outreach, and persists every lead to SQLite. Inference runs locally via Ollama using `mistral:7b-instruct-v0.3-q4_K_M`. All services are containerised with Docker Compose. Live search requires `SERPAPI_KEY` in `.env`.

## Business Context

### Product Domain

HumanMaximizer is an end-to-end HRMS / HCM platform for Indian enterprises. The system qualifies leads specifically for buyers of HRMS software across the following functional areas:

- Payroll processing and statutory compliance (ESIC, PF, PT, CLRA)
- Recruitment and onboarding
- Attendance and shift management
- Leave management
- Employee performance and productivity tracking
- Workforce operations across multi-location and field-force deployments

### HRMS Buyer Targeting

The `ContactFinderTool` (`backend/tools/contact_finder.py`) extracts contacts matching the following decision-maker title patterns:

```
CHRO · Chief Human Resources Officer · Chief People Officer · CPO
VP HR · VP Human Resources · Head of HR · Head of People
HR Director · Director of HR · Director of Human Resources
HR Manager · People Manager
Talent Acquisition Head / Director / VP
Founder · Co-Founder · CEO · Managing Director
```

### Target Industries

The qualification engine (`backend/agents/qualification_agent.py:31–32`) classifies industries as follows:

| Classification | Industries | Score |
|---|---|---|
| Primary | Manufacturing · IT Services · Healthcare · Retail · BFSI | 20 / 20 |
| Adjacent | Automotive · Education · Logistics · Pharma · FMCG · Real Estate | 12 / 20 |
| Unknown | Unknown or missing | 6 / 20 |
| Other | Any other industry | 4 / 20 |

### RAG Knowledge Base

The ingestor (`backend/rag/ingestor.py:24`) crawls `https://www.humanmaximizer.com` (depth=3, max 30 pages). All retrieved RAG chunks used in outreach generation are stored verbatim in `rag_context_used` on every lead record, enabling manual verification that email and LinkedIn content is grounded only in HumanMaximizer product facts.

## Multi-Agent Workflow

### Research Agent

**File:** `backend/agents/research_agent.py`

**Purpose:** Collects and enriches company data from search results, extracts decision makers, and generates an LLM-written research summary grounded in company facts.

**Input** (`LeadState` fields consumed):
- `keyword` — business search term
- `location` — optional city/region filter

**Output** (`LeadState` fields written):

| Field | Type | Description |
|---|---|---|
| `company_name` | str | Company display name |
| `domain` | str | Web domain (e.g. `bharatforge.com`) |
| `description` | str | Raw text from search result or scrape |
| `employee_count` | int \| None | Headcount, None if unavailable |
| `industry` | str | Classified industry label |
| `hq_location` | str | Headquarters city/state |
| `tech_stack` | list[str] | Known HR or ERP tools |
| `decision_makers` | list[DecisionMaker] | name · title · email · linkedin_url · confidence |
| `raw_summary` | str | 150–250 word LLM-generated summary (3 paragraphs: Company Overview · HR Pain Points · HRMS Fit Signal) |
| `growth_signal` | str | One of: hiring_surge · recent_funding · expansion · stable · contracting · unknown |

**Live path:** `SerpSearchTool.run(keyword, location)` → `ScraperTool.scrape(url)` → `ContactFinderTool.find(text, domain)`.  
**LLM call:** `research_summary.j2` → Mistral-7B at temperature=0.2, ctx=8192. Retried twice via `tenacity`. Falls back to template string on failure.

### Qualification Agent

**File:** `backend/agents/qualification_agent.py`

**Purpose:** Scores the lead 0–100 across five HRMS-fit dimensions using a deterministic rule engine, determines pass/fail at threshold 35, and produces an LLM-written two-sentence reasoning.

**Input** (`LeadState` fields consumed):
- `company_name`, `industry`, `employee_count`, `tech_stack`, `decision_makers`, `growth_signal`

**Output** (`LeadState` fields written):

| Field | Type | Description |
|---|---|---|
| `qualification_score` | int | Sum of five dimension scores, 0–100 |
| `score_breakdown` | ScoreBreakdown | Per-dimension integer scores |
| `qualification_reasoning` | str | Two-sentence LLM commentary (strongest reason · biggest risk) |
| `is_qualified` | bool | `True` if `qualification_score >= 35` |
| `status` | str | `"qualified"` or `"disqualified"` |

**Scoring dimensions** (each 0–20 points):

*Company Size Fit*

| Employee Count | Score |
|---|---|
| ≤ 50 | 2 |
| 51–200 | 8 |
| 201–500 | 14 |
| 501–2000 | 18 |
| > 2000 | 20 |
| None (missing) | 8 (mid-range default) |

*Industry Relevance*

| Classification | Score |
|---|---|
| Primary (Manufacturing · IT Services · Healthcare · Retail · BFSI) | 20 |
| Adjacent (Automotive · Education · Logistics · Pharma · FMCG · Real Estate) | 12 |
| Unknown / empty | 6 |
| Any other | 4 |

*Tech Stack Gap*

| Condition | Score |
|---|---|
| Spreadsheet / manual (Excel, Google Sheets) | 18 |
| Legacy HRMS (SAP HR, Oracle HRMS, PeopleSoft) | 12 |
| Empty list / unrecognised stack (unknown — cannot distinguish "no HRMS" from "data unavailable") | 10 |
| Modern HRMS competitor (Darwinbox, Keka, Workday, etc.) | 4 |

> Note: The code conservatively returns 10 for an empty `tech_stack` list because live-mode scraping cannot distinguish "company has no HRMS" from "the page didn't expose tech-stack data". A score of 18 (spreadsheet/manual) requires explicit evidence of Excel or Google Sheets usage in the scraped text.

*Decision Maker Reachability*

| Condition | Score |
|---|---|
| Email + LinkedIn URL | 20 |
| Email only or LinkedIn only | 12 |
| Name present, no contact info | 6 |
| Title found, name unknown | 3 |
| No decision makers found | 0 |

*Growth Signal*

| Signal | Score |
|---|---|
| `hiring_surge` · `recent_funding` · `expansion` | 20 |
| `stable` · `unknown` | 10 |
| `contracting` | 2 |

**Qualification gate:** `is_qualified = (total_score >= 35)`  
**Conditional edge:** score ≥ 35 → `sales_node`; score < 35 → `END` with `status="disqualified"`  
**LLM call:** `qualification_reasoning.j2` → Mistral-7B at temperature=0.1, ctx=4096. Retried twice. Falls back to `"Score: {total}/100. Automated qualification completed."` on failure.

### Sales Agent

**File:** `backend/agents/sales_agent.py`

**Purpose:** Retrieves HumanMaximizer product knowledge from ChromaDB and generates a RAG-grounded cold email and a LinkedIn connection message personalised to the lead.

**Input** (`LeadState` fields consumed):
- `company_name`, `industry`, `employee_count`, `raw_summary`, `decision_makers`

**Output** (`LeadState` fields written):

| Field | Type | Description |
|---|---|---|
| `outreach_email` | OutreachEmail | `{subject: str, body: str}` — 150–220 words |
| `linkedin_message` | str | ≤ 300 characters, no emojis |
| `rag_context_used` | list[str] | Verbatim retrieved chunk texts (audit trail) |

**RAG query construction** (`sales_agent.py:73–79`):
```
"HRMS features for {industry} company [with {employee_count} employees] [{raw_summary[:200]}]"
```

**Email format** enforced by `_parse_email()`:
```
SUBJECT: [subject line]
BODY:
[email body]
```

**LinkedIn limit** enforced by `_enforce_linkedin_limit()`: truncates at last sentence boundary ≤ 300 chars.  
**Hallucination prevention Layer 1:** `cold_email.j2` instructs the LLM to use only the provided RAG chunks and forbids invented statistics.  
**Hallucination prevention Layer 2:** all `rag_context_used` chunks stored on the lead record for human verification.  
**LLM calls:** `cold_email.j2` then `linkedin_message.j2` → Mistral-7B at temperature=0.3, ctx=8192. Each retried twice. Both have template fallbacks.

### Agent Interaction Flow

```
START
  │
  ▼
research_node
  │  Writes: company_name, domain, description, employee_count, industry,
  │           hq_location, tech_stack, decision_makers, raw_summary, growth_signal
  │
  ▼
qualification_node
  │  Writes: qualification_score, score_breakdown, qualification_reasoning,
  │           is_qualified, status
  │
  ▼ _route_after_qualification()
  │
  ├── score < 35 ──► END  (status = "disqualified")
  │
  └── score ≥ 35 ──►
        │
        ▼
      sales_node
        │  Writes: outreach_email, linkedin_message, rag_context_used, status="qualified"
        │
        ▼
       END
```

All state mutations are immutable spreads (`{**state, **updates}`) — each node receives the full prior state and returns an updated copy.

## Lead Generation Pipeline

```
Input
  keyword: str         ← e.g. "manufacturing companies Pune"
  location: str        ← e.g. "Pune, India"
        │
        ▼
Research
  Live mode  → SerpSearchTool  → Google results via SerpAPI (gl=in, hl=en)
               ScraperTool     → httpx GET + BeautifulSoup (blocks LinkedIn, Facebook, Twitter)
        │
        ▼
Enrichment
  ContactFinderTool → regex scan for 18 DM title patterns + email extraction
  LLM summary       → research_summary.j2 → 3-paragraph Mistral-7B output
        │
        ▼
Qualification (deterministic rule engine)
  score = company_size_fit + industry_relevance + tech_stack_gap
          + decision_maker_reachability + growth_signal
  max score: 100
  threshold: 35
        │
        ├── score < 35  → status = "disqualified" → persist → return
        │
        └── score ≥ 35
              │
              ▼
Outreach Generation (RAG-grounded)
  ChromaDB cosine similarity retrieval (k=5) on HRMS query
  cold_email.j2   → Mistral-7B → 150–220 word email (SUBJECT + BODY)
  linkedin_message.j2 → Mistral-7B → ≤ 300 char connection request
        │
        ▼
Ranking
  qualification_score (0–100) is the rank signal
  stored as integer on Lead record; sortable via GET /api/v1/leads
        │
        ▼
Storage
  save_lead() → SQLite leads table → returns UUID
  All JSON blobs (tech_stack, decision_makers, score_breakdown, outreach_email,
  rag_context_used, errors) stored as TEXT columns with JSON serialisation
```

## RAG Implementation

### Ingestion

**File:** `backend/rag/ingestor.py`  
**Trigger:** `python scripts/ingest.py` (or `--refresh` to drop and rebuild)  
**Source:** `https://www.humanmaximizer.com`  
**Crawler settings:**

| Parameter | Value |
|---|---|
| Max depth | 3 |
| Max pages | 30 |
| Timeout per page | 10 s |
| Skipped extensions | `.pdf .jpg .jpeg .png .gif .svg .zip .exe` |
| Stripped tags | `<script> <style> <nav> <footer> <header>` |
| Min text length to index | 200 chars |

### Chunking

**Splitter:** `RecursiveCharacterTextSplitter`  
**Chunk size:** 512 tokens  
**Chunk overlap:** 64 tokens  
**Separators (in order):** `\n\n` → `\n` → `. ` → ` `  

Each chunk carries metadata: `source_url`, `page_title`, `chunk_index`, `ingested_at`, `chunk_id` (MD5 of text).

### Embeddings

**Model:** `nomic-embed-text` via `OllamaEmbeddings`  
**Dimensions:** 768  
**VRAM:** ~270 MB (FP16, 137M parameters)  
**Served by:** same Ollama container as Mistral-7B

### Vector Database

**Engine:** ChromaDB  
**Client:** `chromadb.PersistentClient(path)` — local file-based persistence  
**Collection name:** `humanmaximizer_knowledge` (env-configurable via `CHROMA_COLLECTION`)  
**Persistence:** local directory at `CHROMA_PERSIST_DIR`  
**Upsert:** `collection.add(ids, embeddings, documents, metadatas)` — idempotent on MD5 chunk ID

### Retrieval

**Type:** Cosine similarity search  
**Parameters:**

| Parameter | Value | Effect |
|---|---|---|
| `k` | 5 | Chunks returned to prompt |
| `metric` | cosine | Collection created with `hnsw:space=cosine` |

**Graceful degradation:** `build_retriever()` returns `None` if collection does not exist; `retrieve_chunks()` returns `[]` when retriever is `None`; SalesAgent logs a warning and continues with a single fallback line.

### Hallucination Prevention

**Layer 1 — Prompt constraint (synchronous):**  
`cold_email.j2`: `"Write a personalized cold email using ONLY the facts below. Do not invent statistics."`  
`research_summary.j2`: `"Write a 3-paragraph lead summary using ONLY the facts above. Do not invent statistics, metrics, or contact names."`

**Layer 2 — RAG citation trail (synchronous):**  
`rag_context_used` stored verbatim in every `LeadState` and `Lead` DB record. Every API response includes the exact chunks that grounded the generated content.

**Layer 3 — Self-critique (available, async):**  
`backend/tools/self_critique_tool.py` — `SelfCritiqueTool.check(email_body, rag_chunks)` makes a second Mistral-7B call with `self_critique.j2`, identifies unsupported claims, and emits `hallucination_checks_total{is_grounded=...}` Prometheus counter.

## Model Selection

### Selected Model

`mistral:7b-instruct-v0.3-q4_K_M` served via Ollama.

| Property | Value |
|---|---|
| Parameters | 7B |
| Format | GGUF Q4_K_M |
| Context window | 8,192 tokens |
| License | Apache 2.0 |
| VRAM (Q4_K_M) | ~4.4 GB |

**Why Mistral-7B over alternatives evaluated:**

| Model | Decision |
|---|---|
| Mistral-7B-Instruct-v0.3 | **Selected** — strong instruction-following, Indian-English well-represented, 8k context, Apache 2.0 |
| Llama-3-8B-Instruct | Not selected — larger memory footprint; Mistral equals it on structured JSON output for this workload |
| Phi-3-Mini-4k | Not selected — 4k context too short for multi-paragraph research summaries |
| Gemma-7B | Not selected — restrictive usage policy; less instruction-tuned for B2B sales copy |

**Why 7B is sufficient for this workload:**
- Research summary: ~400 token output, ~2k input
- Cold email: 150–220 word output, constrained by prompt
- Qualification reasoning: 2-sentence output
- Larger models (13B+) require ≥24 GB VRAM, eliminating single-GPU deployment

### Quantization

**Format:** Q4_K_M (GGUF, llama.cpp K-quants)

| Property | Detail |
|---|---|
| Bit depth | 4-bit integer weights (from 16-bit float) |
| K-quants | Mixed precision per tensor block; higher-importance tensors retain more bits |
| M variant | Balanced between Q4_K_S (smaller) and Q4_K_L (larger) |
| Quality vs FP16 | ~97% task performance on instruction-following benchmarks |
| JSON output safety | Q3 and below show measurable JSON corruption in structured tasks; Q4_K_M avoids this |

### Hardware Requirements

| Target | GPU | VRAM | Notes |
|---|---|---|---|
| Development | RTX 3070 / RTX 4060 Ti | 8 GB | Q4_K_M LLM + nomic-embed-text simultaneously |
| Production (recommended) | RTX 4090 / A10G | 24 GB | Higher concurrency or Q8_0 for quality |
| CPU fallback | Any | — | Set `OLLAMA_NUM_GPU=0`; ~10× slower; acceptable for low-volume demos |
| Cloud GPU | AWS g5 (A10G) / Colab T4 | 16–24 GB | T4 (16 GB) handles Q4_K_M comfortably |

GPU is passed to the Ollama container via `docker-compose.yml:16–22` (`deploy.resources.reservations.devices`). Remove that block for CPU-only operation.

## Fine-Tuning Strategy

### When Fine-Tuning is Justified

The current system uses prompting + RAG as the primary approach. Fine-tuning is not the first-line solution because new HumanMaximizer product features are added by re-ingesting the website without any model retraining.

Fine-tuning becomes justified when:
1. The LLM consistently produces malformed `SUBJECT:`/`BODY:` splits requiring additional post-processing
2. Qualification reasoning hallucinates specific compliance regulations (CLRA, ESIC Form 5A, PT returns) not present in RAG context
3. Cold emails include generic placeholder language despite specific RAG context being provided

### Approach: QLoRA with Unsloth

**Why QLoRA over full fine-tuning:** Full fine-tuning of 7B parameters requires 80–120 GB VRAM. QLoRA freezes base model weights and trains small low-rank adapter matrices on top of a 4-bit quantized base. Training VRAM: 8–12 GB on a single GPU. Unsloth accelerates QLoRA training 2–3× via custom CUDA kernels.

**LoRA configuration:**

```python
target_modules = ["q_proj", "k_proj", "v_proj", "o_proj",
                  "gate_proj", "up_proj", "down_proj"]
lora_r = 16          # rank
lora_alpha = 32      # scaling factor
lora_dropout = 0.05
```

### Dataset Format

JSONL in Alpaca format — `{instruction, input, output}` triples:

```jsonl
{
  "instruction": "Write a cold email for a B2B HRMS sale.",
  "input": "Company: Bharat Forge (Manufacturing, 10000 employees, SAP HR). DM: Rajesh Kumar, VP HR. RAG: HumanMaximizer supports multi-plant payroll and CLRA compliance.",
  "output": "SUBJECT: Modernise Multi-Plant HR at Bharat Forge\nBODY:\nDear Rajesh,\n\nBharat Forge's scale across plants means compliance under CLRA and managing multi-site payroll in SAP HR is costly. HumanMaximizer automates statutory filings and consolidates payroll across all plants in one dashboard.\n\nWould you have 15 minutes this Thursday to see a demo tailored for large manufacturers?\n\nBest,\nHumanMaximizer Sales Team"
}
```

**Dataset construction:**
1. Run pipeline in live mode on 200–300 target companies (SerpAPI required)
2. Human sales expert reviews and edits outputs → ground truth
3. Supplement with adversarial examples (micro companies, irrelevant industries) → teaches graceful rejection
4. Target: 300–500 high-quality examples

**Post-training deployment:** Export LoRA adapter → merge into base weights → convert to GGUF Q4_K_M → load via Ollama (drop-in replacement, same API endpoint).

### Latency

`stage_latency` histogram is populated by `MetricsMiddleware` (`backend/observability/middleware.py`) on every HTTP request using `request.url.path` as the stage label. Buckets: `[0.5, 1, 2, 5, 10, 30, 60, 120]` seconds.

### Agent and Pipeline Failures

- Every agent node wraps execution in `try/except`. Exceptions are appended to `LeadState.errors` (a list of strings) and `status` is set to `"error"`.
- The `errors` field is persisted to `errors_json` on the `Lead` DB record.
- `GET /api/v1/leads/{lead_id}` exposes the full `errors` list in the response.
- Structured Python logging (`logging.ERROR`) is emitted with `exc_info=True` for full stack traces.

### Hallucination Monitoring

`SelfCritiqueTool` (`backend/tools/self_critique_tool.py`) performs a second Mistral-7B pass (`self_critique.j2`) comparing every email claim against the retrieved RAG chunks. Result format:

```json
{"hallucinated_claims": ["claim 1"], "is_grounded": false}
```

The outcome increments `hallucination_checks_total{is_grounded="true"|"false"}`.

### Lead Quality Metrics

`lead_qualification_score` histogram records every score at `/api/v1/search` completion. The architecture document (`docs/architecture.md:278`) specifies a monitoring alert when the mean drops below 30, indicating degraded keyword input or search quality.

### LangSmith Agent Tracing

When `LANGCHAIN_API_KEY` is set in `.env`, `configure_tracing()` (`backend/observability/langsmith.py`) enables LangSmith tracing for every LangGraph node execution. Each run captures per-agent token usage, latency, prompt content, and output in a timeline view. Failed nodes surface as error spans with full stack traces.

## Security

### Secret Management

All credentials and configuration are sourced exclusively from `.env` via `pydantic_settings.BaseSettings` (`backend/config.py`). No secrets appear in source code.

| Environment Variable | Type | Usage |
|---|---|---|
| `SERPAPI_KEY` | Optional[str] | Required for live company search via SerpAPI |
| `LANGCHAIN_API_KEY` | Optional[str] | Enables LangSmith tracing; absence disables it |
| `SLACK_WEBHOOK_URL` | Optional[str] | Optional alert channel |
| `DATABASE_URL` | str | SQLite connection string (default: `sqlite+aiosqlite:///./data/leads.db`) |

`.env.example` is committed to the repository. `.env` is gitignored.

### Input Validation

All API request bodies use Pydantic models with type annotations. FastAPI automatically returns 422 Unprocessable Entity for malformed payloads before any application code runs.

### CORS

`CORSMiddleware` is configured with `allow_origins=["*"]` in `main.py`. This is appropriate for a local demo environment.

### Configuration Isolation

`Settings` uses `extra="ignore"` — unknown environment variables are silently discarded, preventing misconfiguration injection.

### Data Persistence

The SQLite database file is stored in `backend/data/leads.db`, mapped to Docker volume `api_data`. All JSON blobs written to TEXT columns are serialised with `json.dumps` and deserialised with `json.loads` — no eval, no exec.

## Known Constraints

1. **Single lead per pipeline invocation.** Each `POST /api/v1/search` call processes one keyword and returns one lead. Batch processing is not implemented.

3. **ContactFinderTool is regex-based.** Decision-maker extraction relies on pattern matching against scraped text. Contact emails are only found when they appear in the raw page content. LinkedIn URLs are not discovered automatically.

4. **ChromaDB must be ingested before outreach generation works.** If `scripts/ingest.py` has not been run, `build_retriever()` returns `None`, `rag_context_used` is empty, and the SalesAgent falls back to a single static line: `"HumanMaximizer is an end-to-end HRMS for Indian enterprises."`

5. **SelfCritiqueTool is not on the critical path.** The hallucination self-critique tool exists and is instrumented but is not invoked automatically during `POST /api/v1/search` or `POST /api/v1/outreach/generate`. It must be called explicitly.

6. **SQLite is the default database.** SQLite is suitable for local demo and single-instance deployments. Under concurrent write load, switching to PostgreSQL via the `DATABASE_URL` environment variable is required.

7. **LLM inference is synchronous.** `pipeline.invoke()` runs on the HTTP request thread. Under slow GPU or CPU-only conditions, the `POST /api/v1/search` endpoint may take 30–120 seconds to respond.

8. **No authentication layer.** The API has no API key, JWT, or OAuth mechanism. All endpoints are publicly accessible on the configured port.
